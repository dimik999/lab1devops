# Пояснение к лабораторной работе №1 (для сдачи и защиты)

Документ объясняет проект «с нуля»: что сделано, зачем каждый файл и как это связано с заданием. Его можно приложить к отчёту или сократить под требования преподавателя.

**Методичка по заданию** — в файле `Лабораторная работа №1. Автоматизация развертывания монолита.pdf` в этой же папке.

---

## 1. Что требовалось по заданию

1. Выбрать простое веб-приложение.
2. Контейнеризовать его (Dockerfile).
3. Настроить CI/CD (GitLab CI или GitHub Actions), чтобы при пуше в ветку:
   - собирался Docker-образ;
   - запускались тесты (если есть);
   - образ публиковался в Container Registry (GitHub Packages / Docker Hub);
   - приложение деплоилось на удалённый сервер (Ansible **или** команды по SSH).

В проекте использованы: **Flask**, **Docker**, **GitHub Actions**, **GitHub Container Registry (ghcr.io)**, деплой по **SSH** на удалённый сервер (опционально) или на **self-hosted runner** (WSL/ПК). Вариант с **Ansible** в методичке указан как альтернатива SSH — в репозитории он не выносился в отдельные файлы, автоматизация деплоя сделана через GitHub Actions и SSH-скрипт на сервере.

## 2. Термины простыми словами

| Термин | Смысл |
|--------|--------|
| **Репозиторий** | Проект на GitHub: код + история изменений. |
| **Git** | Версионирование: коммит, ветка, `push` на GitHub. |
| **Монолит** | Одно цельное приложение (здесь — один сервис Flask), не набор микросервисов. |
| **Docker** | Запуск приложения в **контейнере** — изолированной среде с нужным Python и библиотеками. |
| **Образ (image)** | Шаблон файловой системы и настроек; по нему создают контейнеры. |
| **Контейнер** | Запущенный экземпляр образа (как «живая копия»). |
| **Dockerfile** | Пошаговая инструкция сборки образа. |
| **Registry** | Хранилище образов в интернете. **ghcr.io** — Container Registry от GitHub. |
| **CI** | При изменении кода автоматически: тесты, сборка. |
| **CD** | Автоматическая доставка и запуск (деплой) собранного приложения. |
| **GitHub Actions** | Автоматизация в GitHub: при событии (например `push`) выполняется сценарий из YAML. |
| **Workflow** | Файл `.github/workflows/*.yml` — сценарий CI/CD. |
| **Раннер (runner)** | Машина, на которой выполняются шаги: **ubuntu-latest** в облаке GitHub или **self-hosted** на вашем ПК/WSL. |
| **Секреты (Secrets)** | Пароли и ключи в настройках репозитория; в логах маскируются. |
| **SSH** | Удалённый вход на сервер; по SSH можно выполнять команды на другой машине (в CI это делает экшен, подключаясь к вашему VPS). |

## 3. Общая схема «от кода до сайта»

```
Вы правите код локально
    → git push в GitHub (ветка main или master)
        → GitHub Actions запускает workflow
            → job test: pytest
            → job build-and-push: docker build + push в ghcr.io
            → job deploy (один из двух):
                  • deploy-ssh: по SSH на удалённый сервер → docker pull/run
                  • deploy-self-hosted: на вашем ПК с runner → docker pull/run
```

Локально то же самое можно проверить командами: `pytest`, `docker build`, `docker run`.

## 4. Описание файлов проекта

### 4.1. `app.py` — веб-приложение (монолит)

- Создаётся объект Flask — это веб-приложение.
- Маршрут **`/`** — главная страница, отдаётся HTML-строка с заголовком.
- Маршрут **`/health`** — ответ JSON `{"status": "ok"}` и код 200. Нужен для проверки «сервис жив» (удобно для мониторинга и автоматических проверок).
- **`host="0.0.0.0"`** — приложение слушает все сетевые интерфейсы. В контейнере иначе с хоста часто нельзя достучаться до порта (привязка только к localhost внутри контейнера).
- **`port=5000`** — порт согласован с Docker и деплоем.

### 4.2. `requirements.txt`

Список зависимостей с версиями:

- **flask** — сам фреймворк;
- **pytest** — запуск тестов.

В production-образе в Dockerfile ставится только Flask (образ меньше и проще).

### 4.3. `tests/test_app.py`

- **Фикстура `client`** — тестовый клиент Flask: запросы к приложению **без** запуска реального сетевого сервера.
- **`TESTING = True`** — режим тестирования.
- **`test_index`** — проверка: ответ 200 и в теле есть текст «Лабораторная работа». Используется **`.encode("utf-8")`**, потому что `response.data` — байты, а в Python нельзя писать кириллицу в литерале `b"..."` (только ASCII).
- **`test_health`** — проверка JSON эндпоинта `/health`.

### 4.4. `pytest.ini`

```
[pytest]
pythonpath = .
```

Добавляет **корень проекта** в путь поиска модулей, чтобы строка `from app import app` в тестах находила `app.py` рядом с папкой `tests/`.

### 4.5. `Dockerfile` — основной образ для запуска приложения

Построчно по смыслу:

1. **`FROM python:3.11-slim`** — база: Linux + Python 3.11, урезанный образ.
2. **`WORKDIR /app`** — рабочая папка внутри контейнера.
3. **`COPY requirements.txt .`** — копируется файл зависимостей (отдельным слоем для кэша).
4. **`RUN pip install ... flask==3.0.0`** — ставится только Flask для работы приложения; **`--no-cache-dir`** уменьшает размер; **`--timeout`** и **`--retries`** помогают при медленном интернете до PyPI.
5. **`COPY app.py .`** — копируется код приложения.
6. **`EXPOSE 5000`** — **документация** для человека: приложение слушает 5000. Реальный проброс: **`docker run -p 5000:5000`**.
7. **`CMD ["python", "app.py"]`** — команда при старте контейнера.

### 4.6. `.dockerignore`

Список того, что **не** попадает в контекст сборки (`.git`, виртуальное окружение, лишние файлы). Ускоряет сборку и не тащит мусор в образ.

### 4.7. `.gitignore`

Исключения для Git: `__pycache__`, `.venv`, кэш pytest и т.д., чтобы в репозиторий не попадал мусор.

### 4.8. `.github/workflows/ci-cd.yml` — CI/CD

**Триггер:** `push` в ветки **`main`** или **`master`**.

**Переменная `REGISTRY: ghcr.io`** — адрес GitHub Container Registry.

**Job `test`:**

- Выполняется на **ubuntu-latest** (облачный раннер GitHub).
- Клонируется репозиторий, ставится Python 3.11, зависимости из `requirements.txt`, запуск **`pytest tests/ -v`**.

**Job `build-and-push`:**

- **`needs: test`** — запускается **только если** тесты прошли.
- **`permissions: packages: write`** — токену workflow разрешено публиковать пакеты (образы) в GitHub.
- Логин в **ghcr.io** через `docker/login-action` (логин — пользователь GitHub, пароль — **`GITHUB_TOKEN`**).
- `metadata-action` формирует теги образа (в т.ч. **`latest`** для main/master и тег по коммиту).
- `build-push-action` собирает образ по **`Dockerfile`** и пушит в **`ghcr.io/<владелец>/lab1-app`**.
- Кэш слоёв (**`type=gha`**) ускоряет повторные сборки.

**Job `deploy-ssh`** (удалённый сервер, как в методичке — команды по SSH):

- Включается, если в настройках репозитория переменная **`DEPLOY_VIA_SSH`** равна **`true`** (Settings → Secrets and variables → Actions → **Variables**).
- Нужны секреты: **`SERVER_HOST`** (публичный IP или DNS), **`SERVER_USER`**, **`SERVER_SSH_KEY`**, **`SERVER_PORT`**.
- Действие **appleboy/ssh-action** подключается по SSH и выполняет скрипт на сервере: `docker login` → `docker pull` → остановка/удаление старого контейнера → `docker run` с портом 5000.

**Job `deploy-self-hosted`** (если **не** включён SSH-режим):

- Выполняется на **вашем** раннере (**self-hosted**), например WSL, где запущен агент `run.sh`.
- Те же команды Docker выполняются **локально** на машине с runner. Это нужно, когда **нет** сервера с публичным IP: облако GitHub не может подключиться к внутреннему адресу WSL (172.x).

Тесты в CI выполняются **на раннере GitHub** (`pytest`), отдельный `Dockerfile` для тестов в проекте не используется — так проще поддерживать один образ только для запуска приложения.

1. Репозиторий на GitHub с файлами: `app.py`, `Dockerfile`, `tests/`, `.github/workflows/ci-cd.yml`.
2. Вкладка **Actions** — успешный запуск workflow: **test** → **build-and-push**; при настроенном деплое — **deploy-ssh** или **deploy-self-hosted**.

## 7. Типичные вопросы на защите и короткие ответы

**Зачем Docker?**  
Чтобы окружение было одинаковым на моём ПК, в CI и на сервере; образ один раз собран — везде запускается одинаково.

**Что такое CI/CD в вашем проекте?**  
При push в main/master автоматически гоняются тесты, собирается образ, он публикуется в ghcr.io, затем выполняется деплой (SSH или self-hosted).

**Зачем тесты до сборки образа?**  
Чтобы не тратить время на сборку и не публиковать образ с заведомо сломанным кодом.

**Чем образ отличается от контейнера?**  
Образ — шаблон; контейнер — запущенный процесс на основе образа.

**Зачем `/health`?**  
Быстрая проверка, что приложение отвечает, без разбора HTML.

**Почему `0.0.0.0` в Flask?**  
Чтобы приложение принимало подключения не только с localhost внутри контейнера, а снаружи через опубликованный порт.

**Что такое ghcr.io?**  
GitHub Container Registry — место, куда CI отправляет Docker-образ и откуда сервер делает `docker pull`.

**Зачем два варианта деплоя?**  
SSH — классический удалённый сервер с публичным IP (как в задании). Self-hosted — если такого сервера нет: job выполняется на моём компьютере с установленным runner и Docker.

**Почему нельзя было в SERVER_HOST указать IP WSL 172.x?**  
Это частный адрес в локальной сети; из интернета и с раннера GitHub до него нет маршрута — будет таймаут.

**В методичке упоминается Ansible — у вас его нет в репозитории, это нормально?**  
В задании допускается **либо** Ansible, **либо** команды по SSH. В проекте реализован деплой через **SSH** из GitHub Actions (`appleboy/ssh-action` выполняет на сервере команды `docker login`, `pull`, `run`).

## 8. Краткое резюме одним абзацем (для введения в отчёт)

В работе реализовано простое веб-приложение на Flask с маршрутами главной страницы и проверки работоспособности, покрытое тестами pytest, контейнеризованное с помощью Dockerfile и публикуемое в GitHub Container Registry. Настроен пайплайн GitHub Actions: при пуше в ветку main или master выполняются тесты, сборка и отправка образа в ghcr.io, после чего выполняется деплой либо по SSH на удалённый сервер (при включённой переменной DEPLOY_VIA_SSH и настроенных секретах), либо на локальный self-hosted runner с Docker.
